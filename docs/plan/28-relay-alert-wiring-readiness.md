# Unit 28 - Readiness alert 配線（relay は条件付き）
- 参照: docs/design/detailed/13-ui-relay-backend.md §9.1
- 目的: Unit 31 が定義する **primary launch readiness** を本番運用で実際の alert / 通知へ結びつけ、alert 未配線のまま primary launch しない gate を持たせる。relay `/ws` 関連は、relay-enabled deployment にだけ追加で要求する conditional alert set として扱う。
- 実装対象: **launch-critical な primary readiness / alert** としては、Unit 31 の定義に従い、Unit 29 所有・emit の `ui.snapshot.refresh_failure_total{reason}`、`ui.r2.publish_failure_total`、`ui.r2.publish_failure_streak`、`ui.snapshot.generated_age_ms`、`ui.snapshot.published_age_ms` と、Unit 10 所有・emit の `ui.d1.retention_run_total{result}`、`ui.d1.retention_deleted_rows` を束ねた primary readiness set 用の alert rule と通知経路を整備する。`ui.d1.ingest_failure_total{event_type}` は `/api/history` degraded-path 専用の追加シグナルとして扱い、current-state UI の primary launch gate には入れない。optional `/ws` relay を残す配備に限り、Unit 09 所有の `relay.ws.disconnect_total{reason,handshake_status}`、`relay.ws.connect_duration_ms`、`relay.ws.event_gap_ms` を relay-specific additive alert set として追加し、`handshake_status = auth_rejected` 系を即時ページ・`network` 系を一定時間連続で通知するなどの分類を行う。あわせて retention cron が N 日連続で success を出さない場合の通知、publish_failure_streak が上限値（60 秒待機）に張り付いた場合の通知、配備前 readiness checklist、staging での synthetic failure 注入による alert 疎通確認を行う。
- 完了条件: 採用した監視基盤上で、Unit 31 の primary readiness 定義に対応する freshness 低下系・publish 失敗系・retention 無音停止の **primary alert** が有効化され、通知先・担当・解除条件が定義され、staging で少なくとも 1 回ずつ発火確認してから本番投入できる。`ui.d1.ingest_failure_total{event_type}` は `/api/history` 欠落・遅延を知らせる degraded-path alert として別扱いで運用できる。relay-enabled deployment では追加で構成不備（`auth_rejected`）と一時障害が別ルートで通知される。R2 publish の暴走リトライブレーキが検知できる。
- 依存: Unit 10, Unit 29, Unit 31。relay-enabled deployment で relay-specific additive alert set まで配線する場合に限り Unit 09。
- 検証: alert rule review、synthetic failure drill（R2 PUT 持続失敗、retention cron 無音停止、event ingest 失敗。relay-enabled deployment では加えて `auth_rejected` 持続と切断継続）、notification receipt checklist、pre-production readiness sign-off。
- 非対象: 新規監視 SaaS の選定、本体サーバー側 metric 追加。
