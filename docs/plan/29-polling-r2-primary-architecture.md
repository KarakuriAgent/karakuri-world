# Unit 29 - Polling + R2/CDN primary への切り替え
- 参照: docs/design/detailed/13-ui-relay-backend.md §1, §4, §5, docs/design/detailed/15-ui-application-shell.md §3.3, docs/design/detailed/17-ui-rollout.md §1.1, karakuri-world-ui/README.md
- 目的: UI の current-state 配信を relay `/ws` 主体から切り離し、`/api/snapshot` の fixed-cadence polling と R2/CDN 配信を primary path として成立させる。
- Phase 境界の整理: この Unit が必要とするのは、browser から直接 fetch できる R2 custom-domain snapshot object と、staging ないし同等環境で 5 秒 Edge TTL を観測できる最小構成だけである。`AUTH_MODE=public|access` の両配備手順、R2 CORS / Access cookie pre-seeding、運用時の再現性確立は Unit 22/31 の責務とし、この Unit の hard dependency には含めない。
- 実装対象: publisher cadence 5 秒の正本化、**sub-minute-capable trigger の明文化（Cloudflare Worker `scheduled()` 単独を primary publisher に使わない）**、single-flight refresh / publish queue、`published_at` の再定義、polling 由来の `recent_server_events` cache 方針、quiet period 中も cadence が止まらない不変条件、browser 側 5 秒 polling と合わせた 15 秒 freshness budget の明文化、relay `/ws` を optional accelerator へ格下げする責務整理に加え、**primary path readiness で常用する publisher-side `ui.*` metric set の ownership 整理**（`ui.snapshot.refresh_failure_total{reason}`、`ui.r2.publish_failure_total`、`ui.r2.publish_failure_streak`、`ui.snapshot.generated_age_ms`、`ui.snapshot.published_age_ms` は本 Unit の責務、`ui.d1.retention_run_total{result}` / `ui.d1.retention_deleted_rows` は Unit 10 の retention cron emit を readiness input として参照）をここで確定する。
- 完了条件: relay `/ws` を無効化または切断しても、R2 custom-domain object の `generated_at` / `published_at` が fixed cadence で前進し、`recent_server_events` が polling baseline だけでも構築され、UI の stale / freshness 条件が current-state 表示だけで成立する。加えて primary readiness に使う `ui.*` metrics のうち publisher cadence / freshness 系は relay の有無に依存せずこの Unit の経路だけで増分 / 更新され、D1 retention 系は Unit 10 の emit 点を前提に同じ readiness set へ統合される。Unit 27 は launch blocker ではなく additive option へ移る。
- 依存: Unit 01, Unit 03, Unit 10, Unit 12, Unit 13。
- 検証: fixed-cadence publish test、single-flight / queue test、quiet-period freshness test、relay on/off 比較 smoke test、Phase 1 用最小構成 R2 custom domain の cache hit / TTL 観測、design/doc review。認証モード別の配備再現性は Unit 22/31 で継続検証する。
- 非対象: `/api/history` の新規実装方式、relay 専用 observability の emit 実装（Unit 09 で扱う）、alert rule / 通知配線そのもの（Unit 28 で扱う）。
