# Unit 13 - Snapshot store / polling / stale 判定
- 参照: docs/design/detailed/15-ui-application-shell.md §3, §4, docs/design/detailed/17-ui-rollout.md §3.2
- 目的: UI の正本 state を Zustand store に集約し、5秒 poll・version 単調性・60秒 stale 判定、認証モード別 snapshot fetch、初回 snapshot 取得前の全画面ローディング制御を仕様どおり実装する。
- 実装対象: SnapshotStore、初回表示では `snapshot` 未取得（`snapshot === undefined`）の間だけ shell を全画面ローディングに切り替える gate（初回 fetch error で `snapshot_status = error` の間も維持）、single-flight poll、`AUTH_MODE=public` / `access` に応じた snapshot fetch 切替（`access` では `snapshot_url` への direct fetch に `credentials: 'include'` を付与し、proxy fallback を持たない）、fetch レスポンスの HTTP status → JSON parse → `schema_version` 検証（version 比較前に実施）、同じ fetch validation でも HTTP 非 2xx / parse 失敗は `snapshot_status = 'error'` (transient)、`schema_version !== 1` は `snapshot_status = 'incompatible'` (永続エラー) として分離、same-version success 回復、stale timer、fetch error / incompatible / stale の相互分離、snapshot_url 取得。
- 完了条件: 初回 snapshot 成功までは全画面ローディングのみを表示し、成功後は遅延レスポンスや fetch error があっても snapshot version が巻き戻らず最後の成功描画を維持できる。`schema_version !== 1` は通常の fetch error と別状態として区別され、UI は「観戦 UI の更新が必要です」相当の永続エラー表示に回せる。
- 依存: Unit 12。
- 検証: store unit test、初回は `snapshot` 未取得の間は `snapshot_status: idle/loading/error/incompatible` を問わず全画面ローディングを表示し、初回 fetch error 後も維持されたまま eventual success で通常 shell へ遷移する render test、`AUTH_MODE=public` / `access` で snapshot fetch 先と options が切り替わり `access` では `fetch(snapshot_url, { credentials: 'include' })` を使う test、HTTP 非 2xx / JSON parse failure は `snapshot_status = 'error'` になる test、`schema_version !== 1` は `snapshot_status = 'incompatible'` になり `'error'` とは別状態になる test、version 比較前に `schema_version` を検証する test、out-of-order response test、same-version recovery test、stale timer test、incompatible 状態から次回 poll で valid schema 受信時に ready へ回復する test。
- 非対象: /api/history cache、レイアウト差分、ETag / 304。
