# 12 - マップエディタ

## 1. 管理API拡張

### 1.1 エンドポイント一覧

マップエディタ用に以下のエンドポイントを追加する。APIエンドポイントは管理者認証（`X-Admin-Key`）で保護する。

| メソッド | パス | 認証 | 説明 |
|---------|------|------|------|
| GET | /api/admin/config | Admin | 現在の設定を取得 |
| PUT | /api/admin/config | Admin | 設定を更新（YAML上書き） |
| POST | /api/admin/config/validate | Admin | 設定のバリデーションのみ実行 |
| GET | /admin/editor | なし | マップエディタWeb UI（静的ファイル配信） |

### 1.2 設定取得

```
GET /api/admin/config
```

認証: Admin（08-rest-api.md セクション1.2）。

現在の設定ファイルの内容をJSON形式で返す。

レスポンス (200 OK):

```typescript
interface ConfigResponse {
  config: ServerConfig; // 01-data-model.md セクション6.1 で定義
}
```

`ServerConfig` のすべてのフィールドをそのまま返す。YAML固有の構文（アンカー、エイリアス等）は解決済みのデータとして返す。

### 1.3 設定更新

```
PUT /api/admin/config
```

認証: Admin。

リクエストのJSON本文で設定全体を受け取り、バリデーション後に設定ファイルを上書きする。

リクエスト:

```typescript
interface ConfigUpdateRequest {
  config: ServerConfig;
}
```

レスポンス (200 OK):

```typescript
interface ConfigUpdateResponse {
  status: "ok";
}
```

処理フロー:

1. リクエスト本文を `ServerConfig` として解析
2. 既存のバリデーションロジック（01-data-model.md の各制約）を適用
3. バリデーション成功 → JSON → YAML変換 → ファイル上書き
4. バリデーション失敗 → `400 Bad Request`

エラー:

| ステータス | エラーコード | 条件 |
|-----------|------------|------|
| 400 | `validation_error` | 設定のバリデーションに失敗 |

```typescript
interface ConfigValidationError {
  error: "validation_error";
  message: string;
  details: ValidationDetail[];
}

interface ValidationDetail {
  path: string;   // エラー箇所のパス（例: "map.nodes.1-3.building_id"）
  message: string; // エラーの説明
}
```

### 1.4 バリデーション

```
POST /api/admin/config/validate
```

認証: Admin。

設定を保存せずにバリデーションのみ実行する。エディタの「保存前チェック」に使用する。

リクエスト: `ConfigUpdateRequest` と同一。

レスポンス (200 OK):

```typescript
interface ConfigValidateResponse {
  valid: true;
}
```

エラー: `ConfigValidationError`（セクション1.3と同一形式）。

## 2. バリデーション仕様

### 2.1 バリデーション項目

設定更新時のバリデーションは、既存のサーバー起動時バリデーションと同一のロジックを使用する。

| カテゴリ | 検証内容 | 参照 |
|---------|---------|------|
| スキーマ | 型・必須フィールド・値の範囲 | Zodスキーマ（既存） |
| マップ整合性 | グリッドサイズとノード定義の整合 | 01-data-model.md §1 |
| 建物トポロジ | 壁・ドア・内部の隣接関係、building_id整合 | 01-data-model.md §3.2 |
| NPC整合性 | npc_idとノード定義の整合 | 01-data-model.md §4.2 |
| アクションID一意性 | MapConfig内でaction_idが一意 | 01-data-model.md §5 |
| スポーン地点 | 指定ノードが移動可能ノード | 01-data-model.md §6.6 |

### 2.2 バリデーションの共通化

サーバー起動時の設定バリデーションとマップエディタAPIのバリデーションは**同一の関数**を使用する。バリデーションロジックの二重管理を防ぐため、バリデーション関数はエラー詳細を構造化データ（`ValidationDetail[]`）として返す形にし、起動時バリデーションとAPI双方から呼び出す。

## 3. Web UI

### 3.1 配信方式

マップエディタのWeb UIは、サーバーが静的ファイルとして配信する。

```
GET /admin/editor       → index.html
GET /admin/editor/*     → 静的ファイル（JS, CSS, 画像等）
```

Web UIの静的ファイル配信自体は認証不要とする。認証はAPIリクエスト時（`X-Admin-Key` ヘッダー）にのみ行う。

Web UIは初回ロード時に管理キーの入力ダイアログを表示し、入力されたキーをブラウザのセッションストレージに保持する。以降のAPI呼び出しでは保持したキーを `X-Admin-Key` ヘッダーに付与する。APIが `401 Unauthorized` を返した場合はキー入力ダイアログを再表示する。

### 3.2 技術構成

Web UIはシンプルなSPA（Single Page Application）として構築する。

| 要素 | 技術 |
|------|------|
| グリッド描画 | HTML Canvas |
| UIフレームワーク | 軽量（検討中） |
| API通信 | Fetch API |
| YAML表示 | 非対応（JSON ↔ YAML変換はサーバー側） |

### 3.4 グリッドエディタ

| 操作 | 動作 |
|------|------|
| タイルパレットから選択 → グリッドクリック | 選択したノード種別を配置 |

### 3.5 プロパティパネル

選択中のノードまたはエンティティの詳細を編集するパネル。

**ノード選択時:**

| フィールド | 説明 | 編集 |
|-----------|------|------|

| 種別 | ノードのtype | タイルパレットと連動 |
| ラベル | 表示名 | テキスト入力 |
| 建物ID | 所属する建物（wall/door/interior時） | ドロップダウン |
| NPC ID | 配置されたNPC（npc時） | ドロップダウン |

**建物選択時:**

| フィールド | 説明 | 編集 |
|-----------|------|------|

| 説明 | 建物の説明 | テキストエリア |
| 壁ノード | 壁ノードの一覧 | グリッドから自動集計（読み取り専用） |
| 内部ノード | 内部ノードの一覧 | グリッドから自動集計（読み取り専用） |
| ドアノード | ドアノードの一覧 | グリッドから自動集計（読み取り専用） |
| アクション | 建物アクションの一覧 | 追加・編集・削除 |

**NPC選択時:**

| フィールド | 説明 | 編集 |

| 説明 | NPCの説明 | テキストエリア |
| 配置ノード | NPCのノード | グリッドから自動設定（読み取り専用） |
| アクション | NPCアクションの一覧 | 追加・編集・削除 |

### 3.6 ID変更時の連動更新

`building_id`、`npc_id`、`action_id` は複数箇所から参照されるため、IDの変更時には関連する参照先を連動で更新する。

| 変更対象 | 連動更新先 |
|---------|-----------|
| `building_id` の変更 | 所属するノード（wall/door/building_interior）の `NodeConfig.building_id`、建物内NPCノードの `NodeConfig.building_id` |
| `npc_id` の変更 | 配置ノードの `NodeConfig.npc_id` |
| `action_id` の変更 | 同一建物・NPC内の参照のみ（外部参照なし） |

ノード種別の変更時にも関連エンティティを更新する:

| 変更 | 処理 |
|------|------|
| wall/door/interior → normal | ノードの `building_id` をクリア。対応する `BuildingConfig` の `wall_nodes`/`door_nodes`/`interior_nodes` から除外 |
| npc → normal | ノードの `npc_id`・`building_id` をクリア。対応する `NpcConfig` を削除（確認ダイアログ表示） |
| normal → wall/door/interior | `building_id` の選択を求める（既存の建物から選択 or 新規作成） |
| normal → npc | `npc_id` の選択を求める（既存のNPCから選択 or 新規作成） |

マップ以外のサーバー設定を編集するパネル。タイルパレット下部の「設定」から開く。

| セクション | フィールド |
|-----------|-----------|
| 世界観 | 世界名、説明、スキル名 |
| 移動 | 1ノードあたりの移動所要時間（ms） |
| 会話 | 最大ターン数、インターバル（ms）、受諾タイムアウト（ms）、ターンタイムアウト（ms） |

| スポーン | スポーン地点（グリッドから選択） |
| idle再通知 | 有効/無効、間隔（ms） |

## 4. 設定ファイル操作

### 4.1 読み取り

設定ファイルのパスは環境変数 `CONFIG_PATH`（デフォルト: `./config/example.yaml`）で決定する。API経由の読み取り時は、サーバー起動時と同じパスのファイルを読み取る。

### 4.2 書き込み

設定更新時、同じパスのファイルを上書きする。

書き込み手順:

1. 新しい設定JSONを受け取る
2. バリデーション実行
3. JSON → YAML文字列に変換
4. 一時ファイルに書き出し
5. 一時ファイルを設定ファイルパスにリネーム（アトミック操作）

一時ファイルを経由することで、書き込み中のクラッシュによるファイル破損を防ぐ。

#### YAML再シリアライズの制約

マップエディタ経由で保存した場合、JSON → YAMLの再シリアライズにより以下の情報は保持されない:

- YAMLコメント
- キーの並び順（ライブラリの出力順に従う）
- YAMLアンカー・エイリアス（解決済みデータとして展開される）

手編集との併用を想定する場合はこの制約を運用上考慮すること。

### 4.3 グリッドサイズ変更時の処理

グリッドの行数・列数を変更する場合、範囲外のノード定義の扱いを定義する。

| 操作 | 処理 |
|------|------|
| 拡大（行・列を追加） | 新しいノードは未定義（`normal`扱い） |
| 縮小（行・列を削除） | 範囲外のノード定義を削除。建物・NPCが範囲外ノードを含む場合はバリデーションエラー |

## 5. Web UIの静的ファイル配置

### 5.1 ディレクトリ構成

```
src/
└── admin/
    └── editor/
        ├── index.html
        ├── editor.js
        └── editor.css
```

サーバー起動時に `src/admin/editor/` ディレクトリの静的ファイルを `/admin/editor` パス配下で配信する。
